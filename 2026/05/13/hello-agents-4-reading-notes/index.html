面试速记

一句话概括

这一章主要讲了三种经典智能体构建范式：ReAct、Plan-and-Solve、Reflection。它们分别代表了智能体解决问题的三种思路：边想边做、先规划后执行、执行后反思优化。

背诵版总结

第四章主要介绍了三种经典智能体范式：ReAct、Plan-and-Solve 和 Reflection。ReAct 通过 Thought、Action、Observation 的循环，把模型推理和外部工具调用结合起来，适合搜索、计算、API 调用等需要外部反馈的任务。Plan-and-Solve 将任务分成规划和执行两个阶段，先生成完整计划，再逐步执行，适合逻辑清晰、多步骤推理的任务。Reflection 则通过执行、反思、优化的循环，让智能体能够检查并改进自己的结果，适合代码生成、技术报告和高质量输出场景。三种范式分别强调动态交互、结构化规划和自我优化，在真实 Agent 系统中可以组合使用，形成更稳定、更可靠的智能体架构。

面试表达模板

如果面试官问“你怎么理解 Agent 的经典范式”，可以这样回答：

我理解 Agent 的核心不是简单调用一次大模型，而是设计一个可循环、可反馈、可调用工具的任务执行系统。比较经典的范式有三类：第一类是 ReAct，它通过 Thought、Action、Observation 的循环，让模型边推理边调用工具，适合外部信息查询和 API 调用。第二类是 Plan-and-Solve，它先把复杂任务拆成计划，再逐步执行，适合路径比较明确的多步骤任务。第三类是 Reflection，它让模型对自己的初始结果进行反思和优化，适合代码生成、报告写作等对质量要求高的场景。在实际项目中，这三种范式可以组合使用，例如先规划任务，再用 ReAct 执行工具调用，最后用 Reflection 检查关键结果。

高频面试问题

1. ReAct、Plan-and-Solve、Reflection 的核心区别是什么？

简洁回答：
ReAct 是“思考-行动-观察”的循环，适合需要调用外部工具和动态纠错的任务；Plan-and-Solve 是“先规划、再执行”，适合路径清晰的多步骤任务；Reflection 是“执行-反思-优化”，适合对结果质量要求高、允许多轮迭代的任务。

展开回答：
ReAct 更强调实时交互，模型每一步都会先思考，再调用工具，最后根据工具返回结果继续调整下一步。Plan-and-Solve 则更强调全局计划，先把复杂任务拆成步骤，再按步骤执行。Reflection 的重点不是立即完成任务，而是让智能体对初始结果进行批判和修正，从而提升最终结果的质量和可靠性。

2. ReAct 为什么适合工具调用场景？

ReAct 的核心流程是：

模型先判断当前需要做什么，再决定调用哪个工具，工具返回结果后，模型继续基于观察结果推理。这种方式可以弥补 LLM 自身知识过时、计算不准确、不能访问外部系统等问题。因此，它适合搜索、计算、数据库查询、API 调用等场景。

3. Plan-and-Solve 和 ReAct 有什么不同？

ReAct 是边执行边调整，更灵活；Plan-and-Solve 是先生成完整计划，再逐步执行，更稳定。
如果任务环境变化大、不确定性强，ReAct 更合适；如果任务逻辑清楚、步骤明确，Plan-and-Solve 更合适。

4. Reflection 的价值是什么？

Reflection 的价值在于让智能体具备“自我改进”能力。它不是只生成一次答案，而是先生成初稿，再从质量、逻辑、效率、准确性等角度进行反思，最后根据反馈优化结果。它适合代码生成、技术报告、复杂推理、论文写作等对质量要求高的任务。

5. 三种范式可以组合吗？

可以。一个更完整的 Agent 系统可以采用混合架构：

例如客服智能体可以先规划处理流程，再用 ReAct 查询订单、物流和政策，最后用 Reflection 检查退款判断是否合理、回复语气是否得体。

Hello-Agents 第四章：智能体经典范式构建阅读笔记

1. 本章核心内容

第四章《智能体经典范式构建》从“如何真正构建一个智能体”出发，介绍了现代 Agent 的三种经典范式：

  1. ReAct：Reasoning and Acting
  2. Plan-and-Solve
  3. Reflection

上一章主要讲大语言模型本身，而这一章进一步把 LLM 放到智能体系统中，让模型不只是生成文本，而是能够：

  • 理解用户意图；
  • 拆解复杂任务；
  • 调用外部工具；
  • 根据工具反馈继续推理；
  • 反思并优化自己的输出。

也就是说，本章的重点是从“LLM 能回答问题”过渡到“Agent 能完成任务”。

2. 为什么要学习智能体经典范式？

现在已经有 LangChain、LlamaIndex、AutoGen、LangGraph 等很多成熟框架，为什么还要从零实现 Agent Loop？

原因主要有三点：

第一，直接使用高度封装的框架虽然效率高，但容易只停留在“会调用 API”的层面，不理解 Agent 背后的运行机制。

第二，从零实现可以暴露真实工程问题，例如：

  • 模型输出格式不稳定；
  • 工具调用失败；
  • Action 解析错误；
  • 多轮循环容易死循环；
  • 提示词稍微变化就可能影响行为。

第三，理解底层范式后，才能根据业务需要定制 Agent，而不是只能使用现成框架的默认流程。

因此，本章不是单纯讲概念，而是通过代码实现帮助读者理解 Agent 的核心设计逻辑。

3. 范式一：ReAct

3.1 ReAct 的核心思想

ReAct 是 Reasoning and Acting 的缩写，也就是把“推理”和“行动”结合起来。

它的基本流程是：

其中：

  • Thought：模型的思考过程，用来分析问题、制定下一步计划；
  • Action：模型选择的行动，通常是调用工具；
  • Observation：工具返回的结果，作为下一轮推理的依据。

ReAct 的关键思想是：
思考指导行动，行动结果反过来修正思考。

3.2 ReAct 解决了什么问题？

传统的 Chain-of-Thought 主要让模型“想得更清楚”，但它不能访问外部世界。如果问题涉及实时信息、精确计算或数据库查询，模型容易出现幻觉。

而纯工具调用又缺少推理能力，模型可能不知道什么时候该调用哪个工具。

ReAct 把两者结合起来：

  • LLM 负责推理、判断和规划；
  • 工具负责搜索、计算、查询和执行；
  • Observation 负责给模型提供外部反馈。

因此，ReAct 可以让 Agent 处理 LLM 自身知识库无法覆盖的问题。

3.4 ReAct 的优点

高可解释性

ReAct 会显式输出 Thought、Action 和 Observation，因此开发者可以看到 Agent 每一步为什么这样做。

动态纠错能力强

如果某一步工具返回结果不理想，Agent 可以在下一轮调整搜索词或换一种策略。

适合工具调用

ReAct 天然适合搜索、计算、数据库查询、API 调用等任务，可以突破 LLM 自身知识边界。

3.5 ReAct 的局限

依赖模型能力

如果模型不能稳定遵循格式，或者推理能力不足，就可能生成错误 Action。

执行效率较低

ReAct 通常需要多轮 LLM 调用，任务越复杂，耗时和成本越高。

Prompt 脆弱

提示词格式稍有变化，可能导致模型输出不符合解析规则。

缺少全局规划

ReAct 是“走一步看一步”，灵活但可能缺乏长远计划，复杂任务中可能陷入局部最优。

4. 范式二：Plan-and-Solve

4.1 Plan-and-Solve 的核心思想

Plan-and-Solve 可以理解为：

它把任务分成两个阶段：

  1. Plan：规划阶段
  2. Solve：执行阶段

与 ReAct 每一步都动态思考不同，Plan-and-Solve 会先生成完整的任务计划，然后按照计划逐步执行。

如果说 ReAct 像侦探，一边调查一边调整方向；那么 Plan-and-Solve 更像建筑师，先画好蓝图，再按图施工。

4.3 Solve 阶段

执行阶段由 Executor 负责。
它会逐步读取计划中的每个步骤，并维护历史状态。

执行器的输入通常包括：

  • 原始问题；
  • 完整计划；
  • 已完成步骤和结果；
  • 当前需要执行的步骤。

这样模型在执行当前步骤时，可以知道前面已经得到了哪些结果。

4.4 Plan-and-Solve 的优点

结构清晰

先拆解任务，再执行任务，整个流程更容易理解。

适合多步骤推理

对于数学应用题、流程固定的任务、逻辑路径明确的问题，Plan-and-Solve 比 ReAct 更稳定。

有利于状态管理

每一步执行结果都会被记录下来，后续步骤可以继续使用前面的结果。

4.5 Plan-and-Solve 的局限

计划是静态的

如果一开始生成的计划有问题，后续执行可能会一直沿着错误方向进行。

缺少外部反馈适应能力

相比 ReAct，Plan-and-Solve 不擅长处理环境变化或工具返回异常的情况。

依赖计划质量

最终效果很大程度取决于第一步生成的计划是否合理。

4.6 适用场景

Plan-and-Solve 适合：

  • 多步骤数学推理；
  • 流程固定的任务；
  • 文档处理流程；
  • 数据分析步骤规划；
  • 逻辑清晰的任务拆解。

5. 范式三：Reflection

5.1 Reflection 的核心思想

Reflection 是让智能体具备“自我反思与迭代优化”的能力。

它的核心流程是：

与 ReAct 和 Plan-and-Solve 不同，Reflection 的重点不是“如何执行任务”，而是“如何把结果变得更好”。

5.2 Reflection 的基本流程

Reflection 通常包含三个阶段：

1. 执行

模型先生成一个初始答案或初始方案。

例如生成一段 Python 代码。

2. 反思

模型扮演评审员，检查初始结果的问题。

可能检查：

  • 是否正确；
  • 是否高效；
  • 是否有边界问题；
  • 是否有更优方法；
  • 是否符合要求。

3. 优化

模型根据反思反馈，重新生成更好的版本。

这个过程可以重复多轮，直到反思认为“无需改进”，或者达到最大迭代次数。

5.3 Reflection 中的记忆模块

Reflection 需要短期记忆来记录每一轮尝试和反馈。

原文中设计了一个简单的 Memory 类，用来存储：

  • 每一轮生成的代码；
  • 每一轮反思得到的反馈；
  • 迭代轨迹。

这样后续优化时，模型可以看到之前的尝试和批评意见，不会丢失上下文。

5.5 Reflection 的优点

提升结果质量

它可以把一个“能用”的答案优化成一个“更好”的答案。

增强可靠性

通过自我检查，可以发现逻辑漏洞、事实错误和边界问题。

适合复杂任务

对于代码生成、论文写作、技术报告、复杂规划等任务，Reflection 能显著提升最终质量。

5.6 Reflection 的成本

模型调用成本高

每一轮至少需要一次反思和一次优化，调用次数会明显增加。

延迟更高

Reflection 是串行迭代流程，不适合实时性要求高的任务。

Prompt 设计更复杂

需要分别设计执行、反思、优化等不同角色的提示词。

5.7 适用场景

Reflection 适合：

  • 代码生成；
  • 技术报告；
  • 学术写作；
  • 复杂决策；
  • 高可靠性任务；
  • 需要多轮润色和检查的任务。

6. 三种范式对比

范式 核心流程 主要优势 主要局限 适用场景
ReAct Thought -> Action -> Observation 灵活、可调用工具、能动态纠错 多轮调用成本高，缺少全局规划 搜索、API调用、实时信息查询、工具型任务
Plan-and-Solve Plan -> Solve 结构清晰，适合多步骤推理 计划静态，执行中不易调整 数学推理、流程型任务、逻辑清晰任务
Reflection Execute -> Reflect -> Refine 结果质量高，能自我优化 成本高、延迟高 代码生成、报告写作、复杂推理、高质量任务

7. 如何选择 Agent 范式？

7.1 选择 ReAct 的情况

当任务需要频繁和外部环境交互时，优先选择 ReAct。

例如：

  • 查询实时新闻；
  • 搜索网页资料；
  • 调用数据库；
  • 查询订单信息；
  • 调用计算器；
  • 使用多个 API 完成任务。

核心判断标准是：
任务过程中是否需要不断根据外部反馈调整下一步？

如果是，ReAct 更合适。

7.2 选择 Plan-and-Solve 的情况

当任务本身结构清楚，可以提前拆解成多个步骤时，优先选择 Plan-and-Solve。

例如：

  • 解数学题；
  • 制定学习计划；
  • 拆解数据分析流程；
  • 生成项目实施步骤；
  • 整理文档处理流程。

核心判断标准是：
任务是否可以在执行前先生成一个相对稳定的计划？

如果是，Plan-and-Solve 更合适。

7.3 选择 Reflection 的情况

当任务对质量要求高，并且允许多轮迭代时，优先选择 Reflection。

例如：

  • 生成高质量代码；
  • 优化论文段落；
  • 修改技术报告；
  • 检查复杂方案；
  • 生成正式邮件或商业文案。

核心判断标准是：
一次生成是否不够，需要自我检查和多轮优化？

如果是，Reflection 更合适。

8. 三种范式的组合设计

实际应用中，三种范式并不是互斥的，而是可以组合使用。

一个比较完整的 Agent 架构可以是：

例如构建一个“智能客服退款助手”：

  1. 用 Plan-and-Solve 规划处理流程；
  2. 用 ReAct 查询订单、物流、退款政策；
  3. 用 Reflection 检查退款判断是否合理；
  4. 最后生成用户友好的回复邮件。

这种混合范式比单一范式更接近真实生产系统。

9. 本章对 Agent 开发的启发

9.1 Agent 的核心不是单次回答，而是循环机制

普通 LLM 调用通常是：

而 Agent 的核心是：

因此，Agent 开发的重点不是只写一个 prompt，而是设计一个稳定的循环系统。

9.2 工具调用是 Agent 连接真实世界的关键

LLM 本身只是语言模型，不能直接访问数据库、网页、文件系统或业务系统。
工具让 Agent 具备“行动能力”。

一个成熟 Agent 系统通常需要：

  • 工具注册；
  • 工具描述；
  • 工具选择；
  • 参数解析；
  • 工具执行；
  • 错误重试；
  • 结果反馈。

9.3 输出格式控制非常重要

ReAct 和 Plan-and-Solve 都依赖模型输出结构化内容。
如果模型输出不稳定，后续解析就会失败。

因此，实际开发中可以考虑使用：

  • JSON 输出；
  • Pydantic 校验；
  • Function Calling；
  • Tool Calling；
  • XML 标签；
  • 更严格的 schema。

这比单纯依赖正则表达式更稳定。

9.4 需要防止 Agent 死循环

ReAct 和 Reflection 都可能陷入循环。
因此工程上要设置：

  • 最大步数；
  • 最大迭代次数；
  • 超时机制；
  • 工具调用失败次数限制；
  • 终止条件；
  • 人工接管机制。

9.5 不同任务需要不同 Agent Loop

没有一种范式适合所有任务。
Agent 设计的关键是根据任务特征选择合适的循环方式：

  • 需要实时外部信息：ReAct；
  • 需要稳定拆解流程：Plan-and-Solve；
  • 需要高质量结果：Reflection；
  • 复杂生产系统：三者组合。
